# AGENTS.md

## 适用范围

- 本文件适用于项目根目录及所有子目录。
- 开始任何任务前，必须先阅读：
  - `docs/requirements.md`
  - `docs/implementation-plan.md`
  - `docs/acceptance-criteria.md`
  - 涉及数据库时还必须阅读 `docs/database-design.md`
- 当前活动阶段是 **M1 高保真 UI 原型**。除非用户明确批准进入下一阶段，不得实现真实扫描、数据库持久化或文件修改。

## 技术栈

- 使用 Python 3.12、PySide6 和 SQLite。
- 中文界面使用 Microsoft YaHei UI，英文和数字可使用 Segoe UI。
- UI 优先使用 PySide6 自带组件和统一 QSS，不引入大型第三方 UI 框架。
- 正式数据表格使用 Qt Model/View；原型中的简单模拟表格可以使用 `QTableWidget`。
- 读写文本文件一律使用 UTF-8。
- 不引入不必要的新依赖。

## 产品边界

- 这是本地音乐文件管理工具，不是音乐播放器。
- 禁止擅自增加播放、进度条、音量、封面、在线搜索、账号、推荐、专辑管理或视频功能。
- 不得增加需求中不存在的导航分类和高风险操作。
- M1 中所有操作只能使用内存模拟数据，不能读取、修改、移动或删除用户的真实音乐及歌词文件。

## 架构规则

- UI 层不得直接访问 SQLite 或执行文件操作。
- 所有数据库变更必须通过 repository 层。
- 文件系统是事实来源；SQLite 只保存索引、关系、设置和审计信息。
- 长时间扫描、哈希、复制和移动必须在工作线程运行，禁止阻塞 Qt 主线程。
- 危险操作必须使用“计划 → 预览 → 确认 → 执行 → 校验 → 提交日志”的流程。
- 文件操作失败时不得把数据库记录标记为成功。
- 部分成功必须保存到文件级操作明细，不能只记录一个批次状态。
- 不得硬编码密钥、Token、密码或数据库连接串。

## 文件操作安全规则

- 禁止在没有确认弹窗的情况下删除真实音频、歌词、快捷方式或备份文件。
- 默认导入方式是移动；只有目标文件完成大小与 SHA-256 校验后，才能删除源文件。
- 同名目标文件不得自动覆盖。
- SHA-256 相同判定为重复；目标路径相同但哈希不同判定为内容冲突。
- 删除音乐或歌词时先移动到备份目录，默认保留 7 天。
- 永久清理备份必须再次确认，并展示即将删除的数量和路径范围。
- 恢复备份时遇到同名文件不得覆盖，必须进入冲突处理。
- 删除歌词前必须检查引用关系。
- 音乐移动或重命名后，必须将受管歌单快捷方式更新纳入同一次操作计划。
- 测试真实文件操作时只能使用测试创建的临时目录，禁止使用用户的 Downloads、Music 或其他真实媒体目录。
- 不删除项目文件，除非用户明确要求且已经说明依据、影响和恢复方式。

## 文件名与元数据规则

- 统一文件名格式是 `歌名-歌手.ext`，连接符前后不留空格。
- 优先使用 ID3 的 Title 和 Artist；ID3 不可用时才解析文件名。
- 文件名回退解析使用最后一个半角 `-` 分隔歌名和歌手。
- 无法唯一识别时标记为“待手动确认”，不得自动执行。
- Windows 非法字符替换成对应全角字符或空格，并在预览中明确显示。
- 多歌手统一使用 `、`；英文组合名保持原样。
- 保留原始文件名、原路径、哈希和原始元数据，用于审计与恢复。
- 第一版元数据写入只支持 MP3、FLAC、M4A；其他格式只允许重命名文件。
- 检测到内嵌歌词时优先使用内嵌歌词，并标记“已有内嵌歌词”。

## 开发工作流

1. 开始编码前，先说明当前阶段、准备修改的文件和验证方式。
2. 每次只实现当前阶段内的一个可运行纵向功能切片。
3. 不得采用“先写完所有数据库、再写所有界面、最后补业务”的机械分层开发方式。
4. 优先小步修改，复用已有目录结构、风格和工具链。
5. 遇到未确定的业务规则时先标记风险并请求确认，不得强行猜测。
6. 完成修改后运行与修改范围最相关的测试和静态检查。
7. 汇报：改了什么、为什么改、如何验证、已知风险。
8. 等待用户实际验收；验收通过并得到明确授权后再提交 Git commit。
9. 未提交并验收当前阶段前，不得进入下一阶段。

## 当前测试命令

在项目根目录执行：

```powershell
python -m compileall -q .
python smoke_test.py
```

后续引入 `pytest` 后增加：

```powershell
python -m pytest
```

- 如果测试无法运行，必须说明原因并提供人工验证步骤。
- 不得为了覆盖率编写无意义测试。

## 代码质量

- 类型标注优先，函数职责保持单一。
- 业务状态使用枚举或明确的数据类型，不散落魔法字符串。
- 路径使用 `pathlib.Path` 或专用路径服务处理。
- 数据库事务边界放在应用用例或 repository 协调层，不放在控件事件中。
- 用户可见错误必须给出可理解的信息；详细异常写入操作日志。
- 不自动执行危险命令，不重置生产配置，不清空数据库。
