# 实施计划

## 1. 总体原则

- M1 高保真 UI 原型和 P1 只读扫描、SQLite 索引均已通过最终门禁，当前推进 P2 歌曲信息与安全重命名。
- 每个阶段必须形成可以启动、可以测试、可以由用户验收的纵向功能切片。
- 不采用“先写完所有数据库，再写完所有界面，最后拼业务”的机械拆分。
- 每个产生仓库改动的任务包通过定向测试与监督边界后立即原子提交并推送；阶段最终验收控制发布和进入下一阶段。
- 高风险功能必须先完成规则、预览、测试夹具和恢复方案，再连接真实用户文件。

## 2. M1：高保真 UI 原型

### M1.1 项目规则与可运行基线

目标：固定需求、架构、数据库草案和验收规则，保持应用可以启动、测试可以运行。

交付：

- 项目级 `AGENTS.md`
- `docs/requirements.md`
- `docs/implementation-plan.md`
- `docs/database-design.md`
- `docs/acceptance-criteria.md`
- 可运行的 PySide6 项目骨架

验收：

- 文档之间无范围冲突
- 明确标注 M1 不操作真实文件
- `python -m compileall -q .` 通过
- `python smoke_test.py` 通过

### M1.2 主窗口与音乐库切片

目标：完成主窗口、所有音乐、所有歌词和歌单页面的完整演示流程。

交付：

- Windows 11 浅色视觉系统
- 左侧导航、全局工具栏、表格和状态栏
- 模拟搜索、排序、多选、全选和禁用状态
- 音乐右键菜单和搜索空状态

验收：

- 1200 × 760 与 960 × 600 无重叠
- 表格列与需求一致
- 搜索优先级正确
- 双击音乐行不产生播放行为

### M1.3 文件整理窗口切片

目标：通过模拟数据走通导入、重命名和歌词匹配的预览流程。

交付：

- 导入窗口
- 重命名预览窗口
- 歌词匹配窗口
- 多种成功、等待、重复和冲突状态

验收：

- 每个窗口可独立打开和关闭
- 危险按钮只产生模拟结果
- 内嵌歌词状态优先且不出现导出按钮
- 文件名非法字符处理能在预览中表达

### M1.4 历史、设置和确认切片

目标：完成操作历史、设置、删除音乐、删除歌词和歌单确认流程。

交付：

- 操作历史和文件级明细
- 设置四个分类
- 删除音乐确认弹窗
- 删除歌词引用提示
- 歌单移除提示

验收：

- 删除与歌单移除的文案含义不同
- 备份默认值为 7 天
- 永久清理在界面上要求二次确认
- 所有弹窗的确认和取消顺序一致

### M1.5 视觉检查与交付

目标：完成一轮截图驱动的视觉优化并交付 10 张指定截图。

截图：

1. 所有音乐
2. 所有歌词
3. 歌单
4. 导入
5. 重命名预览
6. 歌词匹配
7. 操作历史

验收：

- 中文不截断，表格不溢出
- 工具栏在最小宽度下不拥挤
- 状态颜色克制
- 界面没有播放器或网页后台特征
- 所有产生改动的 M1 任务包均应在通过门禁后分别提交；最终人工验收用于关闭 M1 和进入 P1

## 3. 后续正式功能纵向切片

用户已授权负责人按阶段门禁连续推进 P1～P8；安全、数据正确性、核心可用性和稳定崩溃问题必须清零，轻微视觉与非关键文案进入 backlog，不阻塞实用功能交付。

### P1：项目基础与只读扫描（已完成）

一次打通：选择目录 → 记住目录 → 扫描音频 → 写入 SQLite 索引 → 主列表显示。

包含：

- SQLite migration 和 repository
- 扫描工作线程和取消机制
- MP3、FLAC、WAV、M4A、OGG、AAC 过滤
- 文件系统与数据库差异标记
- 临时目录集成测试

### P2：歌曲信息与安全重命名

P2 按风险拆成三个独立纵向包；每包通过门禁后立即原子提交，不等待整个 P2 完成：

#### P2-A：只读识别与可编辑预览

- 工作线程读取 Title/Artist；两者必须同时有效，否则整体按文件名最后一个半角 `-` 回退
- 生成可编辑建议、识别来源、冲突和人工确认状态
- `missing` 禁止处理，`external_changed` 必须重新分析
- 不写标签、不重命名、不计算哈希

#### P2-B：安全同目录重命名

- 完整预览和用户确认后执行
- 非法字符、保留名、尾点尾空格和同批冲突校验
- 目标存在绝不覆盖；执行前复核路径、大小和 `mtime_ns`
- 文件系统成功而数据库提交失败时恢复原文件名，并保存文件级结果

#### P2-C：受支持格式元数据写入

- 仅 MP3、FLAC、M4A；WAV、OGG、AAC 继续只重命名
- 在同目录候选副本上写 Title/Artist，回读验证后再安全替换
- 任一步失败时保留或恢复原文件，并保存原文件名、原路径和原始元数据快照

P2 不更新 Windows 快捷方式；音乐移动或重命名后的 `.lnk` 联动统一留到 P5。真实移动导入留到 P6。

### P3：真实搜索、排序与外部变更同步

一次打通：文件系统重新校准 → repository 查询 → 歌名优先搜索 → 歌手回退 → 表头和菜单排序。

包含：

- `QAbstractTableModel`
- `QSortFilterProxyModel`
- 一万条模拟记录性能验证
- `missing` 和 `external_changed` 状态

### P4：歌词扫描与匹配

一次打通：扫描 LRC → 编码识别 → 生成候选 → 自动或人工匹配 → 保存关系 → 处理重复和缺失。

包含：

- 只支持 `.lrc`，移除 `.txt` 歌词支持
- 一对一当前匹配
- 内嵌歌词优先
- 低置信度禁止自动提交

### P5：歌单快捷方式

一次打通：创建歌单 → 选择音乐 → 创建 `.lnk` → 列表显示 → 外部修改重新同步。

包含：

- 同一首音乐重复添加自动跳过
- 移除歌单项不删除音乐
- 音乐移动或重命名后的快捷方式更新
- 损坏快捷方式检测

### P6：安全导入

一次打通：扫描源目录 → 重复和冲突分析 → 完整预览 → 复制临时目标 → 大小和 SHA-256 校验 → 原子落位 → 删除源文件 → 写入操作日志。

必须满足：

- 目标校验前绝不删除源文件
- 同名不同哈希绝不覆盖
- 单项失败不阻止其他安全项目完成
- 崩溃后能识别未完成操作

### P7：删除、备份、恢复和撤销

一次打通：选择文件 → 显示影响范围 → 移动到备份 → 数据库记账 → 恢复或到期清理。

包含：

- 删除歌词前检查引用
- 默认保留 7 天
- 永久清理二次确认
- 恢复同名冲突保护
- 最近一次完整导入撤销

### P8：稳定性和 Windows 可执行程序交付

包含：

- 无权限、长路径、文件占用和磁盘空间不足处理
- 数据库迁移和备份
- 125% 与 150% DPI 验证
- 源码运行文档
- 使用固定 Python 与依赖环境构建 Windows `.exe`
- 验证可执行文件从独立发布目录启动，资源路径和用户数据目录正确
- 创建并验证桌面快捷方式指向最终 `.exe`
- 私人本机 EXE 不上传或分享；如需对外分发，先完成依赖许可证兼容、对应源码与构建材料门禁，或更换不兼容依赖

## 4. 每阶段固定流程

1. Codex 说明准备实现的切片、文件和验证方式。
2. 负责人依据已授权的产品边界确定范围；发现明显错误或需要扩大高风险权限时直接说明并暂停该项。
3. 只实现当前切片，并保持应用随时可运行。
4. 运行相关测试、静态检查和人工验证。
5. 汇报修改文件、验证结果和已知问题。
6. 产生仓库改动的任务包通过门禁后立即原子提交并推送；纯测试或设计任务不创建空提交。
7. 阶段自动化、独立复核和实际运行证据用于关闭当前阶段和控制发布。
8. 当前阶段门禁与授权条件满足后再进入下一阶段。
