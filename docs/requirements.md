# 乐库整理助手产品需求

## 1. 产品目标

乐库整理助手是面向 Windows 11 的本地音乐文件管理工具，用于整理用户下载的音频、LRC 歌词和歌单快捷方式。

核心任务：

1. 扫描本地 Downloads 或用户选择的目录。
2. 将音频和歌词安全移动到统一的 Music 目录。
3. 将音乐文件名规范为 `歌名-歌手.ext`。
4. 管理音频、内嵌歌词和外部 LRC 的匹配关系。
5. 使用 Windows 快捷方式管理歌单，不复制音乐文件。
6. 提供操作历史、备份、恢复和设置。

## 2. 明确不做的功能

本产品不是播放器，不提供：

- 音乐播放、进度条、播放控制和音量控制
- 专辑封面、歌词滚动播放和视频功能
- 在线搜索、在线账号、推荐功能
- 专辑、流派、歌手详情或播放队列管理

## 3. 已完成范围：M1 UI 原型

M1 只实现：

- PySide6 中文浅色界面
- 导航切换、搜索、排序和表格展示
- 单选、多选、全选及按钮启用状态
- 右键菜单、下拉菜单和弹窗开关
- 内存模拟数据和模拟操作结果
- 窗口缩放及 10 个指定截图

M1 不实现：

- 真实文件扫描、复制、移动、删除或备份清理
- SQLite 持久化和数据库迁移
- SHA-256、ID3、LRC 解析或写入
- Windows `.lnk` 创建和更新
- 真实恢复、撤销或文件系统监听

所有危险按钮必须明确表现为原型演示，不得触碰真实用户文件。

### 3.1 已完成范围：P1 只读扫描与索引

P1 打通用户选择目录、后台只读扫描、SQLite 索引和主列表展示，但按小任务包逐步接入：

- 只枚举用户明确选择根目录下的 MP3、FLAC、WAV、M4A、OGG、AAC 文件。
- 文件系统仍是事实来源；SQLite 只保存索引、扫描会话和差异状态，UI 不得直接访问 SQLite。
- 开发和自动化测试只使用 `TemporaryDirectory` 创建的目录夹具与数据库，不扫描真实 Downloads、Music 或其他用户媒体目录。
- P1 不读取音频内容、不计算 SHA-256，不复制、移动、重命名、删除文件，不写入标签、歌词或 Windows 快捷方式。
- 递归扫描必须在工作线程运行并支持协作式取消，不能阻塞 Qt 主线程。
- 磁盘缺失或外部变化只标记 `missing` 或 `external_changed`，不得静默删除索引记录。

### 3.2 已完成范围：P2 歌曲信息与安全重命名

P2 只处理用户显式选择且已经由 P1 建立索引的音频，并拆成三个可独立验收的纵向切片：

- P2-A：在工作线程只读解析 Title 和 Artist；只有两者同时有效才采用标签，否则整体按文件名最后一个半角 `-` 回退；生成可编辑预览，全程不修改媒体。
- P2-B：用户完整预览并确认后，仅在源文件同一目录执行安全重命名；目标存在时绝不覆盖，数据库提交失败时恢复原文件名。
- P2-C：仅支持 MP3、FLAC、M4A 元数据写入；先写同目录候选副本并回读验证，再安全替换，失败时恢复原文件。WAV、OGG、AAC 只允许重命名。
- `missing` 文件禁止分析或执行；`external_changed` 必须重新分析并再次确认，不能沿用旧预览。
- P2 不删除、不跨目录移动、不执行真实导入、不创建或更新 `.lnk`；快捷方式联动属于 P5，移动导入属于 P6。
- 开发、自动化测试和独立验收涉及媒体读写时，只能使用 `TemporaryDirectory` 中由测试创建的夹具。

### 3.3 已完成范围：P3 真实搜索、排序与外部变更同步

- 所有音乐正式列表改用 Qt Model/View，直接消费 repository 提供的不可变资产快照，UI 仍不得直连 SQLite。
- 搜索严格保持“歌名优先；存在歌名结果时不混入歌手结果”，英文忽略大小写，结果同分时保持原列表顺序。
- 支持表头稳定排序并保留明确排序状态；一万条本地索引在自用环境中应保持可交互。
- 重新扫描继续以文件系统为事实来源，把缺失和指纹变化标为 `missing` / `external_changed`，不静默删除记录。
- P3 不新增标签写入、跨目录移动、删除、哈希或 Windows 快捷方式操作；测试仍只使用 `TemporaryDirectory`。

### 3.4 当前开发范围：P4 歌词扫描与匹配

- 只扫描用户显式选择根目录下的 `.lrc`，不把 `.txt` 当歌词；目录边界、reparse 与取消规则沿用 P1。
- 读取 LRC 时识别 UTF-8（含 BOM）、GBK 与 Big5 等常见编码，分析前后文件字节和时间不变。
- 根据规范歌名/歌手生成候选；内嵌歌词优先，外部 LRC 当前匹配保持一对一。
- 高置信度可进入待提交列表，低置信度、重复和冲突必须人工确认，不得自动写入关系。
- UI 不直连 SQLite；歌词资产、当前关系及历史变更通过 repository 保存，不静默删除外部变化记录。

## 4. 运行环境与窗口

- Python 3.12
- PySide6
- Windows 11
- 中文浅色主题
- 默认窗口：1200 × 760
- 最小窗口：960 × 600
- 支持最大化和自由缩放
- 左侧导航固定约 200～220 px，主内容随窗口扩展

## 5. 信息架构

### 5.1 主窗口

主窗口由以下区域组成：

1. 左侧导航栏
2. 顶部全局工具栏
3. 页面标题和搜索操作区
4. 中央文件表格
5. 表格底部状态信息

### 5.2 左侧导航

- 音乐库
  - 所有音乐
  - 所有歌词
- 歌单
  - 搜索歌单
  - 新建歌单
  - 我喜欢的
  - 粤语
  - 通勤
  - 怀旧
  - 古巨基

歌单搜索只匹配歌单名称。不得增加播放器控制区、未匹配分类或封面。

### 5.3 全局工具栏

按钮顺序：

1. 导入
2. 重命名
3. 匹配歌词
4. 操作历史
5. 设置

五个按钮均使用纯文字，静止状态的视觉层级保持一致。鼠标悬浮时文字约增大 10%，但按钮外框尺寸和位置不变；按下时背景变灰；释放或移出后恢复静止状态。此规则仅适用于顶部全局工具栏，不改变主操作、危险操作、导航和筛选等其他按钮的语义样式。

## 6. 页面需求

### 6.1 所有音乐

表格列严格为：

1. 选择框
2. 歌名
3. 歌手
4. 时长
5. 格式
6. 大小
7. 歌词状态

歌词状态：未检查、已匹配、可能匹配、未匹配、冲突、已有内嵌歌词。

无选择时，“添加到”和“删除”禁用。右键菜单只有：打开所在文件夹、重命名、重新匹配歌词。双击不执行播放或其他操作。

### 6.2 搜索

1. 去除输入首尾空格，英文忽略大小写，保留中文。
2. 优先匹配歌名。
3. 只要存在歌名结果，就不展示歌手匹配结果。
4. 没有歌名结果时才匹配歌手。
5. 按完全一致、开头匹配、连续包含、模糊匹配排序。
6. 匹配度相同时按原列表顺序。
7. 可以使用 100～150 ms 防抖。

无结果显示“没有找到匹配的音乐”。

### 6.3 所有歌词

表格列严格为：选择框、歌名、歌手、格式、大小、歌词状态，不显示时长。

删除歌词时，正式版本需要：

- 检查是否被音频引用
- 移动到备份目录
- 默认保留 7 天
- 提供恢复能力

### 6.4 歌单

歌单表格沿用音乐表格。从歌单移除只删除快捷方式，不删除真实音乐文件。外部修改歌单目录后，正式版本应能重新识别和同步。

### 6.5 导入窗口

- 当前“导入”入口仍打开只读扫描窗口，只包含用户选择的扫描文件夹，不包含目标文件夹，也不移动文件。
- 当前音频扫描扩展名为 MP3、FLAC、WAV、M4A、OGG、AAC；扫描结果写入索引并刷新主列表。
- 完整的音频/歌词导入窗口、目标文件夹、移动方式和校验流程属于 P6；P6 目标文件校验成功前不得删除源文件。

### 6.6 重命名预览

- 只处理尚未规范化的音乐文件
- 显示原文件名、建议文件名、识别来源和状态
- 建议文件名可编辑
- 执行前必须完整预览
- “同步 Title/Artist 元数据”默认开启；只对 MP3、FLAC、M4A 生效，用户可取消后仅执行安全重命名

识别规则：

1. Title 和 Artist 同时存在且非空时采用标签结果。
2. 任一标签缺失时，不拼接半标签结果，整体按文件名最后一个半角 `-` 回退解析。
3. 无法唯一识别时必须人工确认。
4. Windows 非法字符替换为全角字符或空格，并在预览中显示。
5. 多歌手使用 `、`，英文组合名保持原样。

### 6.7 歌词匹配

- 使用“未匹配”和“重复与冲突”标签页
- 左侧音频列表，右侧歌词候选和匹配信息
- 支持自动高置信度匹配、手动选择、取消匹配、批量处理和标记忽略
- 外部 LRC 当前匹配采用一对一
- 检测到内嵌歌词时优先使用内嵌歌词，并标记“已有内嵌歌词”
- 内嵌歌词不提供导出按钮

### 6.8 操作历史

记录导入、重命名、删除、歌单和歌词匹配操作。每条记录应能显示文件级成功、失败、路径和原因。

只有最近一次满足以下条件的完整导入可以撤销：

- 导入批次状态成功
- 尚未撤销
- 相关文件未发生外部修改
- 目标文件哈希仍与完成导入时一致

### 6.9 设置

设置分为路径、重命名、备份和数据维护。

- 备份默认保留 7 天
- 可选 7 天、15 天、30 天、永久保留
- 永久清理前必须二次确认
- 重命名前显示预览强制开启且不可关闭

## 7. 正式版本数据与文件规则

- 文件系统是事实来源，SQLite 是索引、关系、设置和审计层。
- 数据库与磁盘不一致时，以磁盘为准，并将记录标记为 `missing` 或 `external_changed`。
- SHA-256 相同判定为重复文件。
- 目标路径相同但 SHA-256 不同判定为内容冲突，不得覆盖。
- 默认导入使用移动；目标文件完成大小和 SHA-256 校验后才能删除源文件。
- 音乐移动或重命名后的受管歌单快捷方式更新属于 P5；P2 不提前写入 `.lnk`。
- 保留原始文件名、原路径、哈希和原始元数据。
- 第一版 Title/Artist 元数据写入支持 MP3、FLAC、M4A；其他格式只重命名文件。
- 开发阶段支持 Python 源码运行；完整功能通过最终门禁后制作 Windows `.exe`，并创建指向该可执行文件的桌面快捷方式。
